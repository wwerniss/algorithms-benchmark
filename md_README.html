Algorithms Benchmark & Visualization

Проект являє собою застосунок для демонстрації роботи різних класичних алгоритмів та структур даних. Проект чітко поділений на дві частини: незалежну алгоритмічну бібліотеку на чистому C++17 (STL) та графічний інтерфейс (GUI) на основі фреймворку Qt5.

Для тестування використовується Google Test (GTest).

‍🤖 Примітка: Документація та Doxygen коментарі в коді були створені за допомогою штучного інтелекту Google Gemini.

Технології

  • Мова: C++17 (використання STL, розумних вказівників std::unique_ptr та лямбда-функцій).
  • GUI: Qt5 (Qt Widgets)
  • Збірка: CMake
  • Тестування: Google Test
  • Документація: Doxygen

Структура проекту

  • core/ — "Ядро" алгоритмів та структур даних. Бібліотека, повністю незалежна від Qt (Header-only або статична бібліотека). Включає класи алгоритмів і реалізацію патернів проектування (GoF).
  • app/ — Виконуваний файл GUI. Відповідає лише за відмальовування масивів, графів, дерев і прив'язку логіки до елементів інтерфейсу (кнопки, повзунки).
  • tests/ — Юніт-тести GTest, які перевіряють, що, наприклад, QuickSort і MergeSort коректно сортують масиви, а дерево AVL залишається збалансованим.

Реалізовані алгоритми

  1. Сортування: Quick Sort, Merge Sort, Bubble Sort, Selection Sort, Insertion Sort.
  2. Алгоритми на графах: Пошук у ширину (BFS), Пошук у глибину (DFS), Алгоритм Дейкстри (Dijkstra) для пошуку найкоротшого шляху.
  3. Дерева: Бінарне дерево пошуку (BST), Збалансоване АВЛ-дерево (AVL Tree).

Патерни проектування (GoF)

Архітектура проекту була спеціально спроектована з активним застосуванням патернів для досягнення максимальної гнучкості та розширюваності коду:

  1. Strategy (Стратегія) Використовується для алгоритмів сортування (ISortStrategy) і пошуку шляхів на графах (IPathfindingStrategy), надаючи єдиний інтерфейс (наприклад, sort()). Дозволяє замінювати алгоритм (QuickSort, MergeSort тощо) прямо під час виконання без зміни клієнтського коду.
  2. Template Method (Шаблонний метод) Застосований у ISortStrategy, де базовий клас реалізує загальну логіку (сповіщення спостерігачів до початку і після закінчення), а специфічну логіку сортування делегує абстрактному методу doSort(), який реалізується вже в конкретних алгоритмах.
  3. Observer (Спостерігач) Алгоритми сортування наслідують IObservable, а SortingWindow (інтерфейс) є IObserver. Коли масив видозмінюється (відбувається перестановка елементів), алгоритм викликає notifyObservers(), тим самим відправляючи свій поточний стан UI. Це дозволяє відв'язати графіку від алгоритму. Відтепер ядро нічого не знає про віджети.
  4. Decorator (Декоратор) Оболонка SortTimerDecorator "обгортає" будь-який алгоритм ISortStrategy, не змінюючи його суті, і додає заміри часу виконання (через std::chrono). В кінці виконання ми можемо стягнути з декоратора тривалість роботи алгоритму і показати її на UI.
  5. Factory Method (Фабричний метод) Клас SortFactory використовується для створення об'єктів алгоритмів за їхнім ім'ям (рядком). Фабрика відразу ж автоматично "обгортає" створений алгоритм в SortTimerDecorator, спрощуючи створення "готового до використання" алгоритму.
  6. Command (Команда) & Memento (Зберігач) Застосовується для реалізації "Машини Часу" під час анімації сортування. Коли Observer отримує сповіщення про зміну стану, він створює StepCommand (Команду), яка всередині себе містить знімок стану масиву (Memento) "до" та "після" (класи SortStateMemento).
    • execute() — відтворити крок сортування.
    • undo() — відкотити крок назад. Це дозволяє користувачеві в GUI робити "Undo/Redo" (або Play/Pause), покроково переміщуючись в історії алгоритму.
  7. Builder (Будівельник) Використовується для покрокового конструювання складних графів (GraphBuilder), дозволяючи гнучко додавати вершини, з'єднувати їх лінійно або випадковим чином. Це робить ініціалізацію графа в UI-коді значно чистішою.
  8. Facade (Фасад) Застосований у системі бенчмаркінгу (SortBenchmarkFacade). Надає єдиний, простий інтерфейс для запуску масштабного тестування продуктивності всіх алгоритмів, приховуючи складну логіку інстанціювання стратегій і декораторів.
  9. State (Стан) Використовується для управління станами головного вікна сортування (SortingWindowStates.h). Завдяки класам IdleState, RunningState та PausedState, вікно автоматично вмикає або вимикає потрібні кнопки інтерфейсу без складних перевірок if/else, що робить логіку UI кристально чистою.
  10. Visitor (Відвідувач) Дозволяє додавати нові операції до структур даних дерев (BST, AVL) без зміни їхнього вихідного коду. Інтерфейс ITreeVisitor та клас TreeSerializationVisitor успішно обходять вузли та виконують над ними дії, дотримуючись принципу "Open/Closed".
  11. Singleton (Одинак) Керівник історії BenchmarkHistoryManager реалізований як глобальний Одинак (Singleton). Забезпечує єдину точку доступу для логування результатів продуктивності (BenchmarkRecord) для всіх підсистем. Дозволяє графічному вікну та бенчмаркам обмінюватись даними без прямої передачі залежностей.
  12. Observer (Спостерігач) — Розширене для журналу Нова вкладка HistoryWindow підписана на BenchmarkHistoryManager. Кожен раз, коли завершується будь-який алгоритм (сортування, пошук шляху), істория розсилає сповіщення. HistoryWindow миттєво реагує та відмальовує нові результати, що повністю позбавляє від необхідності регулярного оновлення (polling).

Збірка та запуск 🛠

Клонуйте репозиторій, після чого перейдіть у корінь проекту:

git clone https://github.com/wwerniss/algorithms-benchmark.git
cd algorithms-benchmark

Для вашої зручності у репозиторії передбачені готові скрипти для збірки та запуску проекту під різні операційні системи у директорії tools/.

‍Примітка: На всіх платформах у вас повинні бути встановлені CMake, C++17 компілятор (GCC/Clang/MSVC) та бібліотека Qt5 (віджети). Фреймворк GTest завантажиться автоматично під час збірки завдяки CMake FetchContent.

🪟 Windows (через скрипти)

Для користувачів Windows існують готові .bat файли у tools/windows/. Переконайтеся, що ви запускаєте їх з командного рядка розробника (наприклад, Developer Command Prompt for VS) або ж у вас коректно налаштовані змінні середовища PATH до компілятора та інструментів Qt5.

:: Збудувати весь проект та виконати тести
tools\windows\build.bat
:: Запустити вже зібраний головний застосунок
tools\windows\run.bat

🐧 Linux (через скрипти)

Для Linux існують розроблені автоматичні bash-скрипти у папці tools/linux.

# Надайте права на виконання скриптам (виконується одноразово)
chmod +x tools/linux/build.sh
chmod +x tools/linux/run.sh
# Зібрати весь проект та автоматично запустити тести
./tools/linux/build.sh
# Запустити головний застосунок після збірки
./tools/linux/run.sh

🍎 macOS (через Makefile)

# 1. Створення директорії та генерація Makefile
mkdir build && cd build
# Якщо Qt встановлено через Homebrew, обов'язково вкажіть шлях:
cmake -DCMAKE_PREFIX_PATH=$(brew --prefix qt5) ..
# Або: cmake -DCMAKE_PREFIX_PATH=/opt/homebrew/anaconda3 ..
# 2. Збірка проекту
make -j4
# 3. Запуск застосунку
./app/AlgorithmsApp
# 4. Запуск автоматичних тестів
ctest --output-on-failure
